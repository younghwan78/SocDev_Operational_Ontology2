# 디지털 트윈 데모 — 시간·프로세스·가정 실험 진행 가이드

> 흐름: **2주 반입 스토리 → 프로세스 신호 → 시점 재구성(as-of) → 가정 실험(what-if) → KPI 시계열**

[데모 스토리 4장면](demo-story.md)이 "위험 발견 → 결정 근거"의 **현재 상태** 흐름을
보여준다면, 이 데모는 시스템의 **디지털 트윈** 면모를 보여줍니다:

- 지도는 **현재**를 보여준다.
- 트윈은 그 현재가 **어떻게 만들어졌는지**(버전 이력·프로세스 신호),
  **그 시점엔 뭘 알고 있었는지**(시점 재구성),
  **달라지면 어떻게 되는지**(가정 실험)까지 답한다.

| 장면 | 기능 | 답하는 질문 |
|---|---|---|
| 1 | 프로세스 신호 (재개·단계 건너뜀) | 이 이슈의 진행이 프로세스답게 흘러왔는가? |
| 2 | 시점 재구성 (as-of) | 지난주 보고 시점에 시스템은 뭐라고 했는가? |
| 3 | 가정 실험 (what-if) | 이 가정이 사실이라면 지도가 어떻게 변하는가? |
| 4 | KPI 시계열 | 이 KPI는 어느 방향으로 움직이고 있는가? |

모든 장면이 결정론입니다 — 같은 데이터, 같은 입력이면 항상 같은 답이 나오고,
모든 판정은 근거 참조를 동반합니다.

## 준비 — 데모 데이터 반입 (2주 스토리)

이 데모의 핵심 기능들은 **데이터의 변화**를 먹고 삽니다. 시드 직후처럼 변화가
없는 DB에서는 재개·건너뜀·시점 차이가 보이지 않으므로, 준비 단계에서 2주에
걸친 반입을 재연합니다. `samples/`에 데모 CSV 3개가 준비되어 있습니다.

### (권장) 데모 전용 DB에서 시작

운영 DB를 건드리지 않고 언제든 처음부터 재현할 수 있습니다.

```bash
# PostgreSQL에 데모 DB 생성 (환경에 맞게 — 예: docker exec <컨테이너> psql -U <user> -d postgres -c "CREATE DATABASE soc_demo")
export SOC_ONTOLOGY_DSN=postgresql://<user>:<password>@127.0.0.1:5432/soc_demo
uv run python -m backend.cli.main db-init
uv run python -m backend.cli.main db-seed
# 백엔드/프론트엔드는 평소대로 실행 (README 참조)
```

### 반입 2배치 실행

데이터 반입 화면에서 파일을 업로드해도 되고(매핑: **개발 이슈** / **KPI 관측**),
CLI로 넣어도 됩니다:

```bash
# ── 배치 1: "W52 주간 반입" ──
uv run python -m backend.cli.main ingest-file --file samples/demo_twin_w52_issues.csv --mapping issues
uv run python -m backend.cli.main ingest-file --file samples/demo_twin_w52_kpi.csv --mapping kpi_observations

# ⏱ 배치 1이 끝난 "지금 시각"을 메모하세요 — 장면 2에서 이 시각을 입력합니다.
#    시점 재구성 입력은 분 단위이므로, 1분 이상 기다린 뒤 배치 2를 넣습니다.

# ── 배치 2: "W53 주간 반입" ──
uv run python -m backend.cli.main ingest-file --file samples/demo_twin_w53_issues.csv --mapping issues
```

### 반입이 만드는 스토리

| 대상 | 배치 1 (W52) | 배치 2 (W53) | 시스템이 잡아내는 것 |
|---|---|---|---|
| 이슈 "UHD60 KPI 시나리오 DDR 대역폭 마진 목표 미달" | 미해결·심각도 높음으로 등록 (ISP·MIF, 목표 W54) | 분석·조치 기록 없이 곧장 **종결** | **단계 건너뜀** + 검증 없는 종결 경고 |
| 이슈 "resume 직후 오디오 경로 handoff 실패 재발 보고" | **종결** 상태로 반입 (검증 테스트 없음) | 필드 재현 확인 — **미해결로 재개** | **재개(역행)** 신호 |
| KPI 관측 (dou_power) | U W26 1278mW, V W20 1452mW 추가 | — | KPI 시계열이 반입만으로 자란다 |

모든 반입은 버전 이력(v1, v2, …)으로 기록됩니다 — 장면 1·2가 읽는 것이 바로
이 이력입니다. 지우는 경로는 없습니다.

## 장면 1 — 프로세스 신호 (이슈 분석)

**할 것**: 이슈 분석 화면에서 "resume 직후 오디오 경로 handoff 실패 재발 보고"를
클릭합니다.

**보이는 것**:

- 상단 붉은 배너 — "**재개된 이슈** — 종결 후 다시 열린 이력이 있습니다 —
  재개 — '종결'→'미해결' (v2, 날짜)".
- 하단 **상태 전이 타임라인** — v1 (생성)→종결, v2 종결→미해결에 **역행** 배지와
  판정 문구("'종결'→'접수' — 역행"), 변경된 필드 목록.
- 좌측 목록의 이슈 카드에도 **재개** 배지. 상단 요약 칩 **정체·지연**으로
  재개·정체·지연 이슈만 모아볼 수 있습니다.

이어서 "UHD60 KPI 시나리오 DDR 대역폭 마진 목표 미달"을 클릭합니다:

- 배너 — "이슈가 **종결 상태이지만 검증 테스트가 없습니다**".
- 타임라인 v2에 **단계 건너뜀** 배지 — "'접수'→'종결' — 분석 / 우회 / 해결 단계
  기록 없음".

**전달 포인트**:

- 상태 **값**이 아니라 상태 **변화**가 신호입니다. 스냅샷만 보면 "종결 1건,
  미해결 1건"일 뿐이지만, 이력을 보면 하나는 **수상한 종결**이고 하나는
  **무효가 된 해결**입니다.
- 판정은 점수가 아니라 **사실 서술 + 버전 참조**입니다. 정상 진행(접수→분석→해결→종결)은
  아무것도 표시하지 않습니다 — 변화가 아닌 것은 잡음입니다.
- 프로세스 모델 밖의 상태 값이 들어와도 침묵하지 않고 "미등재 상태"로 드러납니다.

## 장면 2 — 시점 재구성 (위험 지도 · 포트폴리오)

**할 것**: 위험 지도(Project U) 상단의 **시점 재구성** 입력에 *배치 1 직후*
시각(준비 단계에서 메모한 시각)을 넣습니다.

**보이는 것**: **UHD60 Recording KPI** 행이 **높음** — ISP·MIF 셀이 붉습니다.
그 시점의 시스템은 미해결 고심각도 이슈를 알고 있었습니다.

**할 것**: **현재로 복귀**를 누릅니다.

**보이는 것**: 같은 행이 **중간**으로 내려옵니다 — 이슈가 종결됐기 때문입니다.
셀을 클릭하면 판정 근거가 "과거 유사 이슈"로 바뀌어 있습니다. 지도는 종결된
이슈도 패턴으로 기억합니다. 그리고 장면 1에서 본 대로, 그 종결은 **검증 없는
종결**이었습니다 — 이 "중간"을 얼마나 믿을지는 장면 3에서 실험합니다.

**할 것**: 이번엔 *배치 1 이전* 시각을 넣습니다 (데모 전용 DB라면 시드 이후
시각이어야 합니다 — 시드 이전은 모든 객체가 "당시 없음"이 됩니다).

**보이는 것**: 데모 이슈가 아예 없던 세상입니다. 상단 배너의 **당시 없음** 카운트가
데모 객체 수만큼 표시됩니다 — 그 시점엔 존재 기록이 없었음을 숨기지 않습니다.

**배너 읽는 법** (시점 재구성 중에는 항상 표시):

| 카운트 | 의미 |
|---|---|
| 재생 버전 | 이 시점까지의 버전 기록 몇 건을 재생했는가 |
| 캡처 이전 가정 | 버전 이력이 없어 "이력 기록 시작 전부터 존재"로 가정하고 현재 상태를 쓴 객체 수 |
| 근사 | 첫 기록이 '갱신'이라 당시 payload를 가장 이른 기록으로 근사한 객체 수 |
| 당시 없음 | 그 시점에 존재 기록이 없어 제외한 객체 수 |

**전달 포인트**:

- "지난주 보고 때는 분명 위험하다고 했는데?" 논쟁을 데이터로 끝냅니다 — 시스템이
  **그 시점에 알던 상태**를 그대로 재구성합니다 (기록 시각 기준이며, 사건 발생
  시각과는 다른 축입니다).
- 재구성에 가정과 근사가 섞이면 배너에 **개수로 드러냅니다**. 정직하지 않은
  재구성은 하지 않습니다.
- 포트폴리오 화면에도 같은 컨트롤이 있습니다. 변경 영향 분석은 API
  (`GET /api/v1/as-of/change-impact?ts=…`)로 같은 재구성을 지원합니다.
- 시점이 URL에 실리므로, 주소를 복사해 보내면 상대방도 같은 과거를 봅니다.

## 장면 3 — 가정 실험 (what-if)

**할 것**: 장면 1의 DDR 대역폭 이슈 상세에서 **가정 실험** 카드의
**"가정: 이 이슈가 다시 열리면?"** 버튼을 누릅니다.

**보이는 것**:

- 가정 에코 — "이슈 상태 가정: '종결' → '미해결' · 확신도 상한: medium".
- 위험 등급 변화 — **UHD60 Recording KPI: 중간 → 높음** (ISP·MIF 셀), 바뀐 셀의
  판정 근거까지 그대로.
- "가정 기반 재계산 — 실데이터가 아니며 저장되지 않는다"는 고지.

장면 1~2에서 쌓인 의심("검증 없는 종결")이 여기서 정량이 아닌 **지도 언어**로
번역됩니다: *그 종결이 잘못이라면, 위험 지도는 다시 높음이다.*

반대 실험도 보여주세요 — resume 이슈에서 **"가정: 이 이슈가 해결되면?"**:
"위험 등급 변화 없음 (변화 없는 시나리오: 23)". **바꿔도 지도가 안 바뀐다는 것도
답입니다** — 이 이슈 하나가 등급을 쥐고 있는 게 아니라는 사실을 확인한 겁니다.

**API로만 제공되는 가정 2종** (`POST /api/v1/what-if`):

```jsonc
// 아직 없는 이슈를 가정으로 주입 — "이런 이슈가 나오면?"
{"assumptions": [{"kind": "new_issue", "target_id": "whatif_isp_fhd120_readout",
  "title": "FHD120 고속 촬영에서 ISP readout 마진 부족 가정",
  "scenario_ids": ["high_speed_fhd120_recording"], "ip_ids": ["ip_isp"], "severity": "high"}]}
// → High-Speed FHD120 Recording: 낮음 → 높음, 이슈 신호에 "신규 등장" 표기

// 목표 주차 시프트 — "고객 일정이 3주 당겨지면?"
{"assumptions": [{"kind": "issue_week_shift",
  "target_id": "demo_issue_abox_resume_regress_u", "week_delta": -3}]}
// → 이슈 신호 변화: "지연: 아니오 → 예 — 목표 W51 경과 (기준 W54)"
```

**전달 포인트**:

- 가정 실험과 실제 지도는 **같은 판정 룰 하나**를 씁니다 — 실험 결과와 실제가
  어긋나는 일이 구조적으로 불가능합니다.
- 어떤 가정도 저장되지 않습니다. 응답에는 항상 가정 목록이 에코되고, 가정 기반
  판단의 확신도는 medium을 넘지 못합니다.
- 여러 가정을 한 요청에 섞을 수 있습니다(최대 10개) — "이 이슈가 다시 열리고,
  일정도 3주 당겨지면?" 같은 복합 시나리오.

## 장면 4 — KPI 시계열 (시나리오 상세)

**할 것**: 시나리오 상세 "UHD60 Recording with EIS and HDR Review"(uhd60_recording_eis_on)
개요 탭의 **KPI 시계열 — 과제 간 비교** 카드에서 **dou_power** 칩을 선택합니다.

**보이는 것**:

- Project U / Project V **두 시리즈가 한 차트에** — U는 W10 1450mW → W26 1278mW,
  V는 W8 1520mW → W20 1452mW. 준비 단계에서 반입한 W26·W20 점이 이미 포함되어
  있습니다 — 주간 반입만 하면 차트는 스스로 자랍니다.
- 차트 아래 주차×과제 표 — 표가 계약이고 차트는 보조 시각화입니다.
- **추세 배지** — "개선: W10 1450.0mW → W26 1278.0mW — '낮을수록 좋음' 기준 개선".
  KPI 방향 정의에 비춘 **사실 서술**이며 점수가 아닙니다.
- 칩의 숫자(예: `dou_power · 9`)는 관측 건수 — 시나리오의 주요 KPI가 아니어도
  관측이 있으면 선택할 수 있습니다.

API로는 마일스톤 정렬 비교도 가능합니다:
`GET /api/v1/kpi/series?kpi_id=dou_power&scenario_id=…&align_milestone_type=release`
— U는 ES release(W12) 기준 −2주~+14주로 재배열되고, 해당 유형 마일스톤이 없는
과제는 "비정렬"과 사유가 명시됩니다. 절대 주차가 다른 과제끼리 "같은 단계에서
어느 쪽이 빨랐나"를 비교하는 용도입니다.

## 마무리 — 네 질문으로 회수

| 시스템의 약속 | 이 데모에서 |
|---|---|
| 무슨 일이 있었나 | 버전 이력·상태 전이 타임라인 (장면 1) |
| 무엇에 영향을 주나 | 위험 지도 델타 — 과거·현재·가정 (장면 2·3) |
| 누가/무엇이 왜 판단했나 | 모든 배지·등급에 판정 문구와 근거 참조 |
| 어떤 결정·행동으로 이어지나 | "검증 없는 종결을 재검토하라"가 데이터에서 도출 |

## 데모 정리

- **데모 전용 DB를 썼다면**: DB를 지우면 끝입니다. 재현하려면 준비 절부터 다시.
- **공용 DB에 반입했다면**: 데이터 반입 화면(반입 이력)에서 데모 배치 3건을
  **rollback** 하면 데모 객체가 제거됩니다. 단, 버전 이력에는 rollback도
  "철회" 기록으로 남습니다 — 이력은 지워지지 않는 것이 이 시스템의 정직성이며,
  시점 재구성은 철회 기록을 올바르게 건너뜁니다.

## 진행 팁

- 시점 재구성 입력은 **로컬 시간, 분 단위**입니다 — 배치 사이 간격을 1분 이상
  두지 않으면 두 시점을 구분해 입력할 수 없습니다.
- 모든 장면 상태(선택 이슈, 시점, 프로젝트 필터)는 URL에 반영됩니다 — 특정
  장면 주소를 복사해 공유하면 그대로 재현됩니다.
- 다크/라이트 테마 모두에서 차트·배지가 검증되어 있습니다.
