0.Executive Summary

질문결론
개발 프로세스를 트윈의 대상으로 삼는 것이 이론적으로 정당한가 정당하다. Palantir 자체가 Ontology를 "digital twin of the organization"으로 규정하고[1], 프로세스 마이닝 학계(van der Aalst)는 DTO(Digital Twin of an Organization)를 명시적 연구 목표로 선언했으며[9], ISO 23247은 'process'와 'supporting document'를 트윈 가능한 1급 요소로 분류한다[12].
SoC 설계 플로우 트윈의 선행 사례가 있는가 칩/팹 트윈은 성숙했지만 개발 프로세스 자체의 트윈은 2025년 기준 업계 표준이 없다 — 검증 전문가들이 공개적으로 필요성을 제기하는 공백 영역이다[16]. 즉 선점 가능한 주제.
무엇이 '대시보드'와 트윈을 가르는가 양방향성(kinetic layer). 읽기 전용 모델은 트윈이 아니다[12]. Action Type(승인된 상태 변경의 번들)과 소스 시스템 writeback이 있어야 operational twin이다[3].
데이터 모델의 핵심 선택 단일 case-id 이벤트 로그는 SoC처럼 다객체(스펙·IP·이슈·빌드·리그레션)가 얽힌 프로세스에서 convergence/divergence 왜곡을 일으킨다. OCEL 2.0 기반 object-centric 이벤트 모델을 처음부터 채택하라[10][11].
에이전트의 역할 3분리: 스키마를 만지는 에이전트(builder) / 데이터를 읽고 쓰는 에이전트(operator) / 품질을 반박하는 에이전트(critic). Palantir도 MCP 서버를 스키마용·데이터용으로 분리했다[5]. 쓰기는 반드시 Action Type + HITL 승인 경유[4][7].
최대 리스크 기술이 아니라 (a) 트윈 구축 예산이 프로젝트에 없다는 조직 문제[16], (b) Jira 기록 규율에 따라 트윈 충실도가 결정되는 데이터 문제[19].

1.1Palantir Ontology 해부 — 무엇이 'operational'인가

Palantir 공식 문서 기준으로 Ontology의 정의 요소는 다음과 같다. 전부 1차 출처 + 3인 교차검증 통과 항목이다.

2대 구성Semantic elements = object type / property / link type (기존 데이터 소스에서 매핑) · Kinetic elements = action type / function[1]. 공식 용어는 "elements"이며 "semantic layer/kinetic layer"라는 층 명칭은 통용 해석임검증
자기 규정"In many settings, the Ontology serves as a digital twin of the organization" — 조직·프로세스 트윈의 벤더 측 근거[1]검증
본질데이터 모델이 아니라 의사결정 표현 시스템. 데이터·로직·액션·보안 4중 통합[2][6]검증
Semantic layer 논쟁Palantir는 자사 Ontology가 'semantic layer'라는 규정을 명시적으로 거부 — "얇은 semantic layer로는 4중 통합·운영화가 불가능"[2]. dbt/Cube류 metric layer와의 차별점 주장검증
Action Type객체·속성·링크에 대한 편집 묶음(bundle)의 형식 정의. 상태 변경이 raw write가 아닌 거버넌스된 1급 연산. 트랜잭션이며 전 애플리케이션에 전역 일관 반영[3]검증
쓰기 아키텍처원자적 트랜잭션, 대규모 배치 뮤테이션, 스트리밍 쓰기, CDC 지원 — 트윈은 읽기 전용이 아니라 양방향[2]검증
Interfaceobject type 다형성(폴리모피즘) 제공 — 스키마 설계 시 추상 타입 활용 가능[1]단일출처
주의 — 유통되는 오개념 2건. (1) 일부 해설 블로그의 "semantic/kinetic/dynamic 3층" 프레임에서 'dynamic layer'는 저자 창작 용어로, Palantir 공식 문서에는 없다[27]. (2) "action 편집이 object type의 writeback dataset에 저장된다"는 구체 메커니즘 서술은 본 리서치의 3인 검증에서 0-3으로 기각됐다(현행 문서와 불일치, materialization 개념으로 대체). 사내 문서에 인용하지 말 것.

1.2표준·학술 기반 — 어디에 정박(anchoring)할 것인가

표준/체계SoC 개발 트윈에 주는 것검증
ISO 23247 (제조 DT 프레임워크, 2021~)참조 아키텍처와 어휘. 트윈 대상(OME, Observable Manufacturing Element)을 8종으로 분류: personnel, equipment, material, process, facility, environment, product, supporting document — '프로세스'와 '문서'가 1급 트윈 대상[12]. 단, 적용 범위는 제조로 한정되어 개발 프로세스 적용은 범위 외 확장임을 명시해야 함[13]. NIST는 이 프레임워크가 discrete/batch/continuous 등 상이한 프로세스로 특수화 가능한 generic 설계라고 분석[14]단일출처
BFO / ISO IEC 21838 + IOF CoreNIST가 제안한 계층화 전략: 최상위 BFO → 중위 IOF Core → 도메인 → 애플리케이션 온톨로지[12]. 실무 시사점: 사내 온톨로지도 '전사 공통 상위 타입 → 개발 도메인 타입 → 프로젝트별 타입'으로 계층화하면 재사용·연합(federation)이 쉬워짐단일출처
트윈 판별 기준물리(원본) 대상과의 양방향 통신이 모델/대시보드와 트윈을 가르는 기준[12]. 개발 프로세스 트윈에서 '양방향' = Jira/Confluence로의 governed writeback단일출처
DTC Business Maturity Model조직 채택 성숙도 5단계: Passive → Starter → Progressive → Mature → Master. 기술 외 차원(Ambition&Strategy, Leadership, Culture, Operating Model 등) 평가 — 트윈 성숙은 조직 변화 문제[15]단일출처

1.3프로세스 트윈 이론 — DTO와 Object-Centric Process Mining

개발 '프로세스'를 트윈하는 작업의 학술적 본진은 프로세스 마이닝이다. 핵심 논지 사슬:

#논지출처검증
1프로세스 마이닝의 궁극 목표는 DTO(Digital Twin of an Organization) — 개선 액션을 탐색할 수 있는 조직의 미러[9]. Park & van der Aalst는 DTO를 물리 제품 트윈과 구분해 '비즈니스 프로세스 개선 목적의 미러 표현'으로 정의[8]ICPM 2021 / Springer검증
2DTO는 시각화가 아니라 action-oriented process mining으로 운영화된다: 이벤트 데이터의 제약 위반을 감지하고 소스 시스템으로 교정 액션을 자동 발행하는 closed loop — Foundry action type/writeback과 동형의 패턴[8]ICPM 2021검증
3손으로 그린 프로세스 모델은 실제(P2P/O2C조차)를 담지 못한다 — 트윈은 이벤트 데이터에서 마이닝되어야 함[9]Springer 2021단일출처
4실제 프로세스는 다객체 동시성 — 단일 case-id 로그는 convergence(이벤트 중복 계상)/divergence(무관 이벤트 순서 혼입) 왜곡을 낳는다. 이것이 OCPM(Object-Centric Process Mining)의 존재 이유[9][10]van der Aalst단일출처
5OCEL 2.0(2023)이 object-centric 이벤트 데이터의 표준 교환 포맷: E2O(event-to-object)에 더해 O2O(object-to-object) 관계, 동적 객체 속성, 관계 qualifier 지원. SQLite/XML/JSON 배포. Celonis Process Sphere로 상용화[10]OCEL 2.0 spec단일출처
6OCPM2 방법론(2025): PM2를 확장해 Domain Modeling(객체 타입 식별→개념 모델→액티비티 식별→추출 매트릭스→우선순위)과 Log Extraction(구현→추출→검증) 단계를 규정 — Jira/Confluence류 시스템에서 OCED를 뽑는 실무 레시피[11]. 사례 연구에서 소스 시스템 로그가 객체-액티비티 관계를 누락해 자동 검증 단계가 필수였음arXiv 2503.10735단일출처
종합. Palantir 계열(ontology + action)과 프로세스 마이닝 계열(OCED + action-oriented PM)은 독립적으로 같은 결론에 수렴한다: 객체 중심의 의미 모델 + 이벤트 기록 + 거버넌스된 역방향 액션. SoC 개발 트윈은 이 두 계보의 교집합 위에 세우는 것이 이론적으로 가장 안전하다.

3.2이벤트 모델 — OCEL 2.0 스타일

객체 상태의 스냅샷만으로는 프로세스 분석이 불가능하다. 모든 상태 변화를 event(type, time, attrs, objects[])로 축적하되, 하나의 이벤트가 복수 객체를 참조하게 한다(E2O). 객체 간 관계(O2O)와 함께 이것이 OCEL 2.0의 골자[10].

5.3MCP 통합 계층 — 스키마/데이터 분리 원칙

MCP는 AI-외부시스템 연결의 오픈 표준으로, Jira/GitHub/Git/Postgres류 커넥터가 초기부터 제공됐다[24]. 주목할 것은 Palantir의 이중 서버 아키텍처다: Palantir MCP는 온톨로지 스키마(타입)는 수정하되 데이터는 쓸 수 없고, 데이터 읽기·쓰기는 별도의 Ontology MCP(OMCP)로만 허용[5] — 권한 통제는 LLM이 아니라 플랫폼 계층에서 강제한다[5].

사내 적용 규칙. (1) twin-schema-mcp: 스키마 제안·조회 전용, 스튜어드 승인 액션으로만 커밋. (2) twin-data-mcp: object query(스코프 열거식) + action 실행(HITL 기본). (3) 소스 시스템 직접 쓰기 tool은 에이전트에게 노출하지 않는다 — writeback은 항상 L3 Action을 경유.

모델 왜곡case 기반 로그의 convergence/divergence[9][10]OCEL 2.0 채택을 P1 비가역 결정으로 고정

'트윈 워싱' 반론"결국 대시보드 아니냐"판별 기준으로 반박: 양방향 writeback[12] + 결정 lineage[7]가 없으면 대시보드, 있으면 트윈. P2가 그 분기점
