# TAT 측정 체계 — 질문→근거 도달 시간 (Stage 12)

> 원점 문서 §8 성공 기준 대응: "정보 탐색 시간 단축 / RCA 시간 단축 / 변경 영향 분석 개선".
> 데모 질문별로 **기존 수작업 경로**와 **코크핏 경로**의 도달 시간을 비교 기록한다.

## 측정 방법

- **앱 내 로그**: 상단 "데모 스토리" 진입 시 장면별 경과 시간이 자동 기록되고,
  마지막 장면 뒤 요약(장면별 + 합계)이 표시된다 (localStorage `soc_tat_run`).
- **수동 스톱워치**: 아래 기준표의 질문을 실무 리더가 직접 수행하고 시간을 기입한다.
  "도달"의 정의 = **원본 근거(측정/이슈/테스트/문서 참조)를 화면에서 확인한 시점**.
- 수작업 baseline은 참여자가 자신의 현재 방식(위키/JIRA/엑셀/이메일 검색)으로 같은
  질문에 답하는 시간. 착수 전에 baseline부터 측정한다 (학습 효과 방지).

## 기준표 (원점 데모 질문 5종 + 스토리 4장면)

| # | 질문 | 코크핏 경로 | 예상 클릭 수 | 수작업 baseline (기입) | 코크핏 측정 (기입) |
|---|---|---|---|---|---|
| Q1 | UHD60 recording에서 가장 위험한 IP는? | 홈 heatmap → UHD60 행 → 근거 패널 | 1~2 | | |
| Q2 | ISP power issue는 어떤 시나리오에서 반복됐나? | Ask SoC 프리셋 → 인용 카드 | 1~2 | | |
| Q3 | 8K30 추가 시 영향 IP/KPI는? | 변경 영향: MFC/ISP 선택 → 분석 실행 | 3~4 | | |
| Q4 | UHD60 thermal 해결 판단 evidence는? | Ask SoC 프리셋 → 인용 카드 → 근거 탐색 | 2~3 | | |
| Q5 | 지금 risk 최고 시나리오는? | 홈 heatmap 최상단 행 (0클릭 식별) | 0~1 | | |
| S | 스토리 4장면 전체 (위험→원인→영향→근거) | 데모 스토리 모드 | 장면당 1~3 | | |

## 비교표 산출 방법

워크숍 후 아래 표를 완성한다:

| 질문 | 수작업 중앙값 | 코크핏 중앙값 | 단축률 | 비고 (근거 품질 차이) |
|---|---|---|---|---|
| Q1~Q5, S | | | | |

- 단축률만이 아니라 **근거 품질**(원본 참조 도달 여부, 인용 검증 가능 여부)을 병기한다 —
  수작업은 빠르더라도 근거 없는 기억 기반 답변일 수 있다.
- 목표선 (원점 문서 예시): 리뷰 준비 시간 30% 이상 단축 가능성 확인.

## 클릭 3번 규칙 검증 (UI 공통 원칙 4)

모든 화면에서 "주장 → 원본 근거"가 3클릭 이내인지 워크숍 중 무작위 표본으로 확인한다:
heatmap 셀(1) → 근거 패널 항목(2) → 시나리오 상세 근거(3) ✓ 설계 기준.
