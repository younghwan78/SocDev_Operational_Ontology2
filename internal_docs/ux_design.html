동반 산출물: gate_review_twin_prototype.html (클릭 가능한 목업)
화면은 온톨로지(데이터 구조)를 그리는 게 아니라 "이 게이트를 통과할 수 있는가"라는 하나의 결정을 그린다. 세부는 그 결정을 향해 고도(altitude)를 낮추며 점진적으로만 열리고, 모든 화면은 반드시 행동 또는 다음 질문으로 끝난다. 그래프·대시보드는 이 원칙의 결과물이지 출발점이 아니다.

1.진단 — 왜 detail-first / 그래프-홈이 실패하는가

"이거다 싶은 UI를 못 찾았다"는 건 도구 문제가 아니라 정박점(anchor) 문제일 가능성이 높습니다. 대부분의 twin UI가 반복해서 빠지는 5가지 함정:

온톨로지 그래프를 홈에 그린다

노드-링크 hairball은 twin을 만든 사람의 뷰다. 보는 사람에게 우선순위·의미·할 일이 없다. 그래프는 근거를 따라갈 때만 가치가 있다(A4).

모든 것이 대시보드 타일

20개 위젯을 평등하게 나열하면 서사가 없다. "무엇을 먼저 봐야 하나"에 답하지 못한다.

대칭적 상세 (symmetric detail)

모든 객체를 동일 밀도로 보여준다. 실제로는 6개 축 중 1개가 지배 요인인데, 화면은 그걸 말해주지 않는다.

정적 스냅샷

"지금 상태"만 있고 "지난 게이트 대비 무엇이 바뀌었나"가 없다. 주기적 의식(ritual)에서 가장 중요한 정보가 빠진다.

읽기 전용 = 대시보드

화면이 행동으로 이어지지 않으면 그건 twin이 아니라 리포트다. 결정·writeback 경로가 없으면 사람들은 다시 안 온다.
핵심 통찰. Palantir 문서조차 온톨로지를 "데이터가 아니라 결정의 표현"으로 규정한다(레퍼런스 문서 §1.1). 그런데 대부분의 구현은 데이터를 그린다. 정박점을 데이터 구조가 아닌 반복되는 결정 의식(=게이트 리뷰)으로 옮기는 순간, "무엇을 어떻게 보여줄지"가 저절로 정해진다.

2.설계 3원칙

원칙내용이론 근거
P1 · 결정 중심 (Decision-first, not data-first)모든 화면은 "이 게이트를 통과할 수 있는가"라는 단일 질문에 봉사한다. 객체·링크·이벤트는 그 질문에 필요할 때만 등장한다.Endsley SA 모델(지각→이해→투영), Palantir "decision representation"
P2 · 고도별 점진 공개 (Progressive disclosure by altitude)판정(A0)→상황판(A1)→리스크(A2)→선택지(A3)→근거(A4). 위에서 아래로 갈수록 밀도↑·범위↓. 아래층은 위층에서 드릴할 때만 열린다.Shneiderman "Overview first, zoom & filter, details-on-demand"
P3 · 막다른 화면 금지 (So-what / Now-what)어떤 화면도 시각화로 끝나지 않는다. 각 화면은 "그래서 의미는(so-what)"과 "그래서 뭘 하나(now-what)"를 항상 품는다 — 상황판 카드마다 한 줄 해석 + "리스크 보기 →".BLUF(Bottom Line Up Front) 브리핑 원칙

3.Altitude 모델 — 정보 구조의 뼈대

사용자의 직관("전체 상황 → 리스크 → 선택지 → 검토")을 5개 고도로 형식화한 것. 각 고도는 답하는 질문·소요 시간·주 사용자가 다르다. 이 축이 곧 내비게이션이다.

◀ 범위 넓음 · 밀도 낮음
A0
판정 / Posture 3초 · 한눈
지금 GO 가능한가? 한 줄 판정 + 신뢰도 + 지배 차단요인 1개
화면 최상단 고정 배너. "조건부 NO-GO · ISP 커버리지". 임원은 여기서 90% 끝난다.
A1
상황판 (Situation Board) 30초
전체 그림은? 6개 exit 축의 신호등 + 추세 + Δ지난게이트 + 한 줄 해석
"무엇을 먼저 볼지"를 색과 정렬 순서가 대신 결정. 그래프 아님 — 축별 카드. 히어로 화면.
A2
워치리스트 (Risk & Watchlist) 2분
무엇이 위협하나? impact×likelihood 순 리스크 + "방치 시" 투영
축을 클릭하면 그 축의 리스크만 필터링. 각 리스크는 "왜"와 "방치 시 결과"를 품는다.
A3
선택지 워크스페이스 (Decision / Options) 10분
어떤 선택지가 있고 뭘 검토하나? 각 선택지의 what-if: 일정Δ·PPAΔ·잔존리스크·투입
여기서 결정이 내려진다. 선택 = Action 스테이징 → HITL 승인 → writeback. Kinetic layer의 UI.
A4
근거 / 온톨로지 상세 (Evidence) on-demand
근거가 뭔가? 연계 객체 테이블 + 추적성 체인(REQ→IP→TC→Defect)
여기가 그래프·raw 객체가 사는 유일한 곳. 그것도 전체 그래프가 아니라 이 결정에 필요한 이웃만.
범위 좁음 · 밀도 높음 ▶
왜 이게 "detail 한꺼번에" 문제를 푸는가. 세부는 사라진 게 아니라 아래 고도로 이동했을 뿐이다. 사용자는 A0에서 결론을, A1에서 그림을 받고, 필요한 축으로만 내려간다. 관심 없는 5개 축의 raw defect 테이블을 볼 일이 없다. 이것이 인지 부하를 구조적으로 제거하는 방식이다.

4.Persona × Lens 매트릭스 — 하나의 twin, 여러 진입점

혼합형 요구("설계·검증·PM·경영이 같은 twin을 본다")의 답은 앱을 나누지 않는 것이다. 페르소나는 기본 착륙 고도 + 기본 렌즈 + 깊이 상한만 바꾼다. 같은 데이터, 다른 진입점.

페르소나기본 착륙기본 렌즈고도 범위역할
임원A0/A1(요약)전체A0·A1판정과 상위 리스크만. 상세 진입 안 함
아키텍트 (게이트 의장)A1전체A0–A4판정 소유. A1↔A3를 오가며 결정
검증 리드A2커버리지·품질A1–A4검증 축 소유. 홀·defect까지 깊이 드릴
PMA2일정A1–A3블로킹 체인·what-if 중심

렌즈(Lens)는 같은 상황판·워치리스트를 특정 exit 축(커버리지/품질/PPA/일정/Spec)으로 재색칠·재정렬한다. 렌즈를 바꿔도 데이터는 하나 — 관점만 바뀐다. 프로토타입에서 상단 렌즈 세그먼트를 눌러 확인할 수 있다(관련 없는 축은 흐려짐).

설계 규칙. 페르소나 전환이 다른 숫자를 보여주면 안 된다(single source of truth 붕괴). 같은 twin의 같은 상태를, 다른 고도·렌즈·상한으로 볼 뿐이다. 임원과 검증리드가 같은 ISP 커버리지 87%를 본다 — 임원은 A1에서 한 줄로, 검증리드는 A4에서 47개 홀 테이블로.

6.게이트 리뷰 스토리보드 — 60초에서 10분까지

아키텍트가 프로토타입을 여는 순간부터 결정 커밋까지의 실제 여정. 각 단계의 소요 시간이 고도 설계와 맞물린다.

그림 — 판정(A0) → 상황판(A1) → 워치리스트(A2) → 선택지(A3) → (필요 시 근거 A4) → Action 스테이징 → HITL 승인 → 커밋·lineage. 근거 확인은 선택적 분기.

실제 시나리오 (프로토타입에서 재현 가능):

단계화면사용자가 얻는 것
1A0 배너"조건부 NO-GO. 지배 차단요인 1개: ISP 커버리지 87%." — 3초 만에 결론
2A1 상황판6개 축 중 커버리지 red, 나머지 amber/green. Δ 켜면 "커버리지 3주째 정체"가 보임
3A2 워치리스트(커버리지)리스크 3건, 1위 "홀 47개 미해소 · 방치 시 샘플 2주 지연"
4A3 선택지수정(2주, 잔존 Low·추천) / 조건부 waiver(즉시 통과, 잔존 Medium) / 디스코프(-5일 회복, 잔존 High). 각 what-if 비교
5Action → HITL"수정" 선택 → ECO WorkItem 생성 + REQ/Defect 링크 writeback 미리보기 → 승인 → Jira 반영 + 결정 lineage 기록

7.안티패턴 체크리스트 — 다음 구현 전 통과 기준

그동안의 구현이 "이거다" 못 느낀 이유를 역으로 점검하는 리스트. 하나라도 해당하면 정박점을 다시 봐야 한다.

  • 홈 화면이 결론(판정)이 아니라 데이터(그래프·표)로 시작하는가?
  • 첫 화면에서 "무엇을 먼저 봐야 하나"에 답하지 못하는가? (모든 위젯이 평등하면 실패)
  • 6개 축 중 지배 요인 1개를 화면이 짚어주지 못하는가?
  • "지난 게이트 대비 무엇이 바뀌었나"를 볼 방법이 없는가?
  • 어떤 화면이 행동/다음 질문 없이 시각화로 끝나는가? (so-what/now-what 부재)
  • 페르소나 전환이 다른 숫자를 보여주는가? (SSOT 붕괴) — 고도·렌즈만 바뀌어야 함
  • raw 온톨로지 그래프가 드릴 없이 처음부터 노출되는가?
  • 결정을 내려도 소스 시스템으로 writeback·lineage 경로가 없는가? (=대시보드)
  • 상태색이 색만으로 의미를 전달하는가? (아이콘·라벨 동반 필수)
