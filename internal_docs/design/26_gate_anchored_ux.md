# 설계 26 — 게이트 리뷰 정박 UX 재편 (검토 + 전체 개선안)

> 검토 대상: `internal_docs/ux_design.html` (Gate Review UX 설계 근거) +
> `internal_docs/gate_review_twin_prototype.html` (클릭 가능 목업, 2026-07-19).
> 이 문서는 두 산출물의 **충실한 검토**와, 현재 58 구현에 얹는 **전체 개선안**이다.
> 상태: 설계 검토 문서 — 구현 착수는 §8 방향 결정 + 사용자 승인 후.

## 1. 검토 요약 — 두 문서가 말하는 것

### 1.1 ux_design.html (설계 근거)

**진단**: "이거다 싶은 UI를 못 찾은" 것은 도구가 아니라 **정박점(anchor)** 문제.
twin UI의 5가지 반복 함정 — ① 온톨로지 그래프를 홈에, ② 모든 것이 평등한
대시보드 타일, ③ 대칭적 상세(전 객체 동일 밀도), ④ 정적 스냅샷("지난 게이트
대비" 부재), ⑤ 읽기 전용(행동 경로 없음 = 리포트).

**처방 — 3원칙**:
- **P1 결정 중심**: 모든 화면은 "이 게이트를 통과할 수 있는가" 단일 질문에 봉사.
- **P2 고도별 점진 공개(Altitude)**: 판정(A0, 3초) → 상황판(A1, 30초) →
  워치리스트(A2, 2분) → 선택지(A3, 10분) → 근거(A4, on-demand).
  아래층은 위층에서 드릴할 때만 열린다.
- **P3 막다른 화면 금지**: 모든 화면이 so-what(한 줄 해석) + now-what(다음
  행동/질문)으로 끝난다.

**보조 구조**: Persona × Lens — 임원/아키텍트/검증리드/PM은 같은 데이터를
**착륙 고도·렌즈·깊이 상한**만 달리해 본다 (SSOT — 페르소나 전환이 다른
숫자를 보여주면 실패). 인터랙션 문법 4종: 신호등+추세+Δ지난게이트 /
드릴=필터 / 선택=Action 스테이징(HITL·writeback·lineage) / 근거 on-demand.
안티패턴 체크리스트 9항목이 구현 통과 기준.

### 1.2 gate_review_twin_prototype.html (목업)

설계를 1:1로 구현한 단일 파일 앱: A0 판정 배너("조건부 NO-GO · 지배 차단요인
1개: ISP 커버리지"), A1 6개 exit 축 카드(신호등·스파크라인·Δ·so-what·"리스크
보기→"), A2 리스크 카드(impact×likelihood heat + "방치 시" 투영), A3 선택지
what-if 비교(추천 배지) → Action 스테이징 드로어(writeback 미리보기 + HITL
승인 + 결정 로그 FAB), A4 연계 객체 표 + REQ→IP→TC→Defect 추적성 체인,
페르소나/렌즈 세그먼트, Δ토글, 브레드크럼.

### 1.3 검토 평가

**이 진단은 옳다.** 58의 현재 화면 지형에 그대로 적중한다:

- 홈(위험 지도)은 시나리오×IP **매트릭스가 홈** — 데이터-우선이다. "질문이
  곧 메뉴"라고 했지만 실제로는 "무엇을 먼저 봐야 하나"를 사용자가 스스로
  정해야 한다 (함정 ②). 어제 리허설에서 "변경된 게 없는데?"라는 반응이 그
  증거다 — 화면이 지배 요인을 짚어주지 않았다.
- 10개 화면이 사실상 평등한 메뉴로 나열되고, 이번 설계 21~25로 만든 트윈
  표면(연결률·워터마크·게이트·링크 제안)이 **출처 지도·리뷰 센터 구석에
  분산**돼 서사가 없다 (함정 ③).
- "지난 게이트 대비"는 as-of 두 시점 diff로 **엔진은 있는데** 사용자가 시각을
  직접 타이핑해야 한다 — 앵커가 없다 (함정 ④).
- 결정·액션 CSV 왕복이 있으나 화면 동선의 끝이 아니라 리뷰 센터의 부가
  기능이다 (함정 ⑤의 절반).

**핵심 통찰의 가치**: 설계 23으로 만든 마일스톤 게이트 판정이 바로 이 정박점
계약이다 — 우리는 이미 A0의 데이터를 갖고 있으면서 A0 화면이 없다.

## 2. 현재 구현 대응표 — 이미 있는 것 / 부분 / 없는 것

| 목업 요소 | 58 현재 | 판정 |
|---|---|---|
| A0 판정 배너 (GO/NO-GO + 지배 요인 + 신뢰도) | `MilestoneGateReview`(met/not_met/근거) + 연결률(신뢰 한계) + AsOfMeta — **데이터 전부 존재, 화면 없음** | **신규 조립** |
| A1 exit 축 카드 6종 | 게이트 kind 3종(이슈 상한/근거 존재/검증된 종결) + 리뷰 팩 rollup + KPI 시계열 | **부분 — kind 확장 필요** |
| 신호등+스파크라인+Δ | 배지 문법·KPI 시계열 존재, 축 카드에 미조립 | 부분 |
| Δ 지난 게이트 토글 | as-of 두 시점 diff(Y2) 엔진 존재 — **게이트 주차 앵커만 없음** | 부분 (배선) |
| A2 워치리스트 (I×L heat 순) | 위험 지도 정성 등급+근거 — **매트릭스형이지 목록형 아님** | 부분 + **충돌**(§3-C1) |
| "방치 시" 투영 | 없음 | 신규 + **주의**(§3-C2) |
| A3 선택지 what-if 비교 | what-if 워크벤치(가정 2종·저장 세트) + `DecisionOption` 계약(benefits/risks/cost_impact — **미사용**) | 부분 |
| Action 스테이징→HITL→writeback | 결정/액션 CSV 왕복(=현재의 HITL), writeback은 ⑤(Stage 19) | 부분 — CSV가 등가물 |
| 결정 lineage 로그 | Decision + 워터마크(W1) + 감사 기록 | **있음** — 노출만 |
| A4 추적성 체인 | traceability / RCA 7단 체인 / 근거 탐색 / 이슈 상세 | **있음** — 드릴 연결만 |
| Persona × Lens | 역할 7종은 advisory 전용, UI 페르소나 없음 | 신규 (URL 프리셋으로 경량) |
| 브레드크럼 / 드릴=필터 | URL=상태 규약 존재 | 부분 |

**결론**: 이것은 재작성이 아니라 **재정박**이다. 설계 8~25가 만든 부품(게이트
판정·as-of·워터마크·연결률·what-if·RCA·추적성)을 A0→A4 동선으로 재조립하는
일이고, 진짜 신규는 A0 배너, 축 카드 조립, 워치리스트 목록형, 게이트 앵커
Δ, 페르소나 프리셋 정도다.

## 4. 방향 결정 — 홈의 정박점 (사용자 승인 필요)

원점 문서의 5질문 코크핏(현재 홈=위험 지도)과 이 설계의 "게이트 리뷰 정박"은
**대체가 아니라 위계 관계**로 정리할 것을 제안한다:

- **홈 = A0/A1 (게이트 콘솔)**: "다음 게이트를 통과할 수 있는가"가 첫 화면.
  주기적 의식(주간 리뷰·게이트 리뷰)의 착륙점.
- **기존 5질문 화면은 A2~A4의 드릴 목적지로 강등이 아니라 재배치**: 위험
  지도=A2 워치리스트의 지도형 뷰(토글), 이슈 분석/RCA=A4, 변경 영향·what-if=
  A3의 검토 도구, Ask SoC=전 고도의 탈출구(자유 질문).
- 근거: 원점 문서의 목적은 "질문이 곧 메뉴"였고, 게이트 정박은 그 질문들에
  **우선순위와 서사**를 부여하는 상위 질문이다. TAT 단축 목표와도 일치 —
  "무엇을 먼저 볼지"를 화면이 결정하면 탐색 시간이 준다.

대안(보수적): 홈은 위험 지도 유지, 게이트 콘솔을 리뷰 센터의 최상단 모드로.
→ 권고하지 않음: 정박점 문서의 진단(함정 ①②)이 다시 적용된다. 단, 최종
선택은 사용자 몫.

## 5. 개선안 — 재편 패키지 G1~G6

> 공통 원칙: 전부 결정론 파생 뷰 + 기존 부품 재조립. 저장 계약 변경은 G2의
> 게이트 kind 확장(additive)뿐. 각 패키지는 독립 검증 가능.

### G1. A0 판정 배너 + 게이트 콘솔 홈 (핵심, 최우선)

- **게이트 선택**: 과제별 "다음 마일스톤"(week ≥ 현재 기준 최근접, exit_criteria
  보유) 자동 선택 + 드롭다운 전환. 리허설/실데이터에서 마일스톤 주차 매핑이
  없으면 "게이트 미지정" 정직 표기.
- **판정 배너**: `충족 m / 미충족 n / 판정 불가 k` → 한 줄 판정
  ("조건부 미충족 — 지배 요인: 실측 근거 누락"). **지배 요인 = 미충족 기준 중
  결정론 대표 1건** (심각도 필터 이슈 수 최다 → 근거 누락 순 — 룰 명시).
  GO/NO-GO 단어는 쓰지 않는다 — 판정은 조언이지 차단이 아니다(설계 23 원칙).
- **신뢰도 줄**: 연결률(이 게이트 프로젝트의 이슈 연결률) + 데이터 신선도
  (최근 배치 시각) — "이 판정이 못 보는 것"을 배너 자체가 말한다. 어제 혼란
  ("변경 없는데?")의 구조적 수리.
- now-what: [지배 요인부터 보기 → A2] [전체 상황판 → A1].

### G2. A1 상황판 — 게이트 축 카드 + Δ지난 게이트

- **축 = 게이트 kind의 확장** (additive, §6.2 체인). v1 축은 현재 데이터로
  계산 가능한 것만:
  1. 미해결 이슈 (기존 max_open_issues)
  2. 근거 태세 (기존 required_evidence + 근거 사다리 measured/predicted/absent)
  3. 검증된 종결 (기존 verified_closure)
  4. **일정 신호** (신규 kind `schedule_signal`: due_week 경과·마일스톤 주차까지
     잔여 — J3 재사용)
  5. **KPI 예산** (신규 kind `kpi_budget`: KPIObservation 최신값 vs KPIDefinition
     목표 — 시계열이 스파크라인)
  6. 링크 커버리지 (W2 재사용 — "이 게이트 판정의 시야" 메타 축)
  - 커버리지/회귀 축(C6)은 사내 검증 DB 확보 후.
- 카드 문법 = 목업 그대로: 신호등(정성 등급) + 스파크라인(KPI/주차 추이) +
  **Δ지난 게이트**(직전 마일스톤 주차의 as-of와 현재 diff — Y2 엔진에 앵커만
  추가) + so-what 한 줄(결정론 템플릿 + supporting_basis) + "리스크 보기 →".
- 색맹 안전: 아이콘·라벨 동반 (기존 배지 규약 준수 확인).

### G3. A2 워치리스트 — 목록형 위험 뷰 (드릴=필터)

- 위험 지도 판정을 **목록형으로 재표현**: 정성 등급 순 + 근거 수 순 (C1 치환).
  각 행 = 제목 + 등급 배지 + 근거 요약 + **룰 기반 "이대로면" 서술**(C2:
  "게이트 기준 X 미충족 상태 지속", "목표 주차 W21 경과 3주"). 축 카드
  클릭 시 해당 축 필터 (URL=상태).
- 지도형(현 위험 지도)은 토글로 유지 — 매트릭스가 유용한 질문(IP 횡단 비교)이
  따로 있다.
- now-what: 행 클릭 → A3(검토 도구) 또는 A4(근거).

### G4. A3 선택지 워크스페이스 — DecisionOption 활용 + CSV 스테이징

- 휴면 계약 `DecisionOption`(benefits/risks/cost_impact)을 반입 표면으로 살린다:
  이벤트의 `candidate_options`(이미 존재)를 선택지 카드로 렌더 + what-if
  워크벤치 연결(가정 실험) + 관련 근거 태세.
- **스테이징 = 결정 CSV 생성**(C3): "이 선택지로 결정 초안" → 결정 CSV 행
  프리필(선택지·근거 ref) 다운로드 → 반입 센터 → dry-run → 반입 →
  워터마크·lineage 자동 (W1이 이미 함). Stage 19에서 같은 버튼이 writeback으로
  진화 — 동선 설계를 지금 고정하는 것이 이 패키지의 핵심 가치.
- 추천 배지는 달지 않는다 — 결정은 사람이 (§6.3). "검토 참여 권장 역할"만
  (기존 규약).

### G5. A4 근거 — 기존 화면의 드릴 목적지화

- 신규 화면 없음. A2/A3에서의 진입 링크 정비: 이슈→이슈 분석(RCA·타임라인),

- 브레드크럼: A0→A4 경로 상단 고정 (URL=상태 확장).

### G6. Persona × Lens (경량) + 타임라인 간트

- 페르소나 = **URL 착륙 프리셋 4종** (`?view=exec|arch|verify|pm`): 착륙
  고도·기본 필터·표시 깊이만 변경. 별도 권한/데이터 분기 없음 (SSOT).
